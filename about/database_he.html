מקורות מידע

נכון לעכשיו, מאגר הנתונים מורכב ממידע משני מקורות:‏

  • USDA - רוב המידע לגבי דברים "נאים" - מירקות ותבלינים עד לבשר, נלקח מכאן. במקרים בהם המידע נמצא כאן, ערכים ועומסים גליקמיים נלקחים מהאתר Glycemic Index Guide, שגם לוקח מידע מה-USDA.‏
  • foodsdictionary.co.il - במקרים בהם מידע משתנה מאוד בין מקורות שונים, או ה-USDA לא מספק מידע, אני משתמש באתר הזה בשביל למלא את המידע שחסר. האתר הזה גם מספק מידע אמין.‏
  • foodstruct.com - כאשר מדובר במוצר נפוץ ממותג פופולרי (לדוגמה: קולה, ביג-מק) .‏כמעט תמיד יש עליו מידע באתר הזה. נראה שהמידע באתר הזה ,מדויק, וכן אני מרגיש בנח לסמוך עליו/li>

כשנוסף מידע לאתר, הוא כמעט תמיד יהיה מאחד מהמקורות שמוזכרים למעלה. גם אם יש מידע, אם הוא למספיק אמין, לא אשים אותו באתר, ואפילו מעבר לזה - אם מוצר מסויים הוא יותר מדי "גנרי", וחברות שונות שמציעות אותו מספקות ערכים תזונתיים שונים, הוא גם לא יכלל.‏

דוגמאות של מאכלים "גנריים" שלא נלקחו לאתר:‏

  • קורנפלקס - כמעט אף אחד לא אוכל קורנפלקס בכלל בלי סוכר, וחברות שונות מוסיפות כמויות שונות מאוד של סוכר לקורנפלקס שלהם.‏
  • פירות משומרים - כל חברה מעבדת את הפירות המשומרים אחרת, ואחד מהחומרים שבד"כ מוסיפים הוא סוכר.‏
דוגמאות של מאכלים "גנריים" שנלקחו לאתר:‏
  • סוגי פסטה שונים - פסטות מחבילה כמעט תמיד נעשות מסוג קמח אחד (סמולינה, דורום, לבן) + מים/ביצים, שלשניהם השפעה מינימלית על הערכים התזונתיים של המאכל.‏
לסיכום - אם לאותו מאכל מחברות שונות יש ערכים תזונתיים שונים מאוד, הוא לא יכלל תחת שם המאכל לבד, ויוצמד אליו שם החברה.‏

איך משתמשים מספקים מידע

כאשר משתמש מספק לאתר מידע, זה אוטומטית כותב קוד בשביל האתר, ,ומודיע לי שמשתמש ביקש להוסיף מידע לאתר ויחד עם זאת נותן לי אופציה להכניס את הקוד שנכתב אוטומטית לאתר.‏
אני (המפתח) עובר על המידע והקוד שנשלח לי, בודק אם הוא הגיוני ומדיוק, לפי התנאים שהזכרתי למעלה. אם המידע שסופק טוב, אני משלב את הקוד שצורף איתו אל תוך האתר. אם המידע שסופק לא טוב, אני סוגר את הבקשה לסיפוק מידע, ולא קורה כלום.‏

When a user makes a contribution, it automatically writes "potential" code for the website, and alerts me of a request to write this code to the website.
I Go over the data, and see if it is supported in any of the trusted sources, if available. If not - I try to look for data in other sources, to see if they support the request's data.

If the data is supported, I manually allow the data to be merged into the website (may sound complicated, takes basically two clicks 😅), update the leaderboards (if the user who provided the data gave their name) and the website online is updated.

If the data is partially supported, i fix what needs fixing, and do the steps above.

If the data is not supported, I reject the request to add the code to the website, and basically nothing happens.

Different Identifiers

You may have noticed some cells in the database contain some extra information, in the form of additional signs/replacement words. These are listed below:

  • ~ - One of three reasons (ordered from most common to least common):
    • The food contains a very broad range of data, and the most supported & common value was chosen
    • A source other than the USDA (or any of website's other trusted sources) was used because of a lack of data.
    • Too many reliable sources conflicted. In which case, a sensible average is chosen
  • < There is no accurately averageable data, but there is a "clear" upper bound for it.
  • No Data - No reliable data was found for this food item. Important to note - Even if one column has no data, that doesnt necessarily mean that the other columns have innacurate/estimated data. If they do, a ~ is used.
